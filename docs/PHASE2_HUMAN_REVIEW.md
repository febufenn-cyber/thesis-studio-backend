# Phase 2 — Human Review and Editing Workspace

Phase 2 turns the Phase 1 canonical thesis into a safe human correction workflow. It is a structured thesis editor and review cockpit, not a browser clone of Microsoft Word.

## Product promise

The operator can see how Robofox interpreted the manuscript, compare any imported block with the immutable original, correct content or structure through validated commands, resolve evidence and formatting issues, preview the authoritative rendered document, and restore earlier states.

## Core invariants

1. **Every edit is a command.** The browser never replaces the full chapter collection during normal editing.
2. **Every command is version-bound.** A stale browser tab receives HTTP 409 before it can overwrite newer work.
3. **Every command has a server-generated inverse.** The latest command can be undone exactly. Cross-chapter moves and batches use a whole-document inverse.
4. **Undo never rewrites history.** Undo and redo create additional append-only commands.
5. **Older history uses snapshots.** Periodic, manual, pre-restore and initial-editor checkpoints are immutable.
6. **Stable IDs survive editing.** Editing or converting a block preserves its UUID; inserted/duplicated blocks receive new UUIDs.
7. **Presentation does not mutate authorship.** Margin, font, casing and pagination remain format-profile concerns.
8. **Trust is invalidated on the server.** Editing a cited block removes its exact citation-resolution records; editing a linked quotation marks that quote unverified; chapter approval returns to `needs_review` after content changes.
9. **Blocking findings cannot be dismissed.** Deterministic integrity, metadata, citation, quotation and preservation failures must be fixed at the source. Only warnings/information and eligible import judgments may be acknowledged.
10. **Preview fidelity has two levels.** Structural preview is immediate and approximate. Authoritative preview is generated by the real DOCX → LibreOffice PDF pipeline and is cached by document/profile version.
11. **Exports are visibly stale.** A ready export records the exact document version that produced it and the workspace labels it stale after any later edit.
12. **Canonical JSON has its own schema version.** Application migration `v2 → v3` is independent from Alembic relational migrations.

## Workspace layout

### Structure sidebar

- Front-matter sections and review states
- Chapters, headings and review states
- Deep links to chapter/block IDs
- Unresolved-only filter
- Structured search

### Structured editor

- Paragraph runs preserve italics individually
- Heading levels 2/3
- Block and verse quotations
- Explicit review/evidence markers
- Insert, edit, convert, move, duplicate, split, merge, delete
- Restore block from immutable import
- Chapter statuses: imported, needs review, in progress, reviewed, approved, locked

### Context sidebar

- Original imported block vs current block
- Linked sources and exact citation decisions
- Registered quotation and verification state
- Review findings anchored to the block UUID
- Recent commands affecting the block

### Review inbox

Persistent review items are synchronized from deterministic verification using a stable fingerprint of:

- rule;
- stable chapter/block/source/quote location;
- active revision for import-specific findings.

Array indexes are excluded from identity. If an underlying problem disappears, the item becomes superseded. If it recurs, the same item reopens with a resolution-history event.

### Metadata and front matter

Candidate/submission placeholders are editable. Institution-controlled certificate, declaration, title-page and contents structure remain profile-driven. Free-text editing is limited to appropriate sections such as acknowledgements and abbreviations.

### Registry

Source forms are generated from source kind, while preserving raw imported entries and unknown fields. Editing verified source fields invalidates verification unless the operator explicitly re-verifies against the source. Quotations remain exact, location-bound registry records.

### Preview

- Fast structural preview: immediate, approximate, not pagination-authoritative.
- Authoritative PDF preview: durable background job through the real renderer and LibreOffice.
- Cache key: project + document version + resolved profile version.
- A changed document never reuses an older preview as current.

### History

- Append-only command timeline
- Latest-only safe undo and immediate redo
- Manual and automatic checkpoints
- Structural snapshot comparison
- Restore creates a pre-restore checkpoint first
- Completed exports show their originating document version and stale state

## Command API

`POST /projects/{project_id}/editor/commands`

```json
{
  "command_type": "update_block_text",
  "payload": {
    "block_id": "...",
    "runs": [
      {"text": "Achebe’s ", "italic": false},
      {"text": "Things Fall Apart", "italic": true}
    ]
  },
  "expected_document_version": 42,
  "client_request_id": "browser-generated-id"
}
```

The client request ID makes autosave retry idempotent. A successful response returns the new document version, command record, changed IDs and trust invalidations.

Important command families:

- `update_block_text`
- `update_block`
- `insert_block`
- `delete_block`
- `duplicate_block`
- `move_block`
- `change_block_type`
- `split_block`
- `merge_blocks`
- `add_marker`
- `restore_block_original`
- `update_chapter`
- `set_chapter_status`
- `update_metadata`
- `reorder_chapters`
- `reorder_front_matter`
- `batch`

Internal restore commands are rejected from normal API requests.

## Undo and snapshots

`POST /projects/{project_id}/editor/commands/{command_id}/undo`

Only the latest command can be undone safely. Older changes require restoring a snapshot because later edits may depend on them.

`POST /projects/{project_id}/editor/commands/{undo_command_id}/redo`

Redo is available only immediately after that undo.

Snapshot endpoints:

- `POST /projects/{project_id}/editor/snapshots`
- `GET /projects/{project_id}/editor/snapshots`
- `GET /projects/{project_id}/editor/snapshots/{id}/compare`
- `POST /projects/{project_id}/editor/snapshots/{id}/restore`

Automatic checkpoints occur every 25 document mutations, plus the first editor command and before snapshot restoration.

## Review and readiness API

- `GET /projects/{project_id}/review-items`
- `GET /projects/{project_id}/review-items/{id}`
- `PATCH /projects/{project_id}/review-items/{id}`
- `GET /projects/{project_id}/readiness`

Readiness percentages are transparent completion ratios across structure, metadata, citations, quotations, formatting and unsupported-content handling. The percentage is informational; one blocking finding still means the project is not ready.

## Editor navigation API

- `GET /projects/{project_id}/editor/structure`
- `GET /projects/{project_id}/editor/chapters/{chapter_id}`
- `GET /projects/{project_id}/editor/front-matter/{entry_id}`
- `GET /projects/{project_id}/editor/blocks/{block_id}/context`
- `GET /projects/{project_id}/editor/search?q=...`

The editor loads one chapter at a time so large theses do not render thousands of blocks into the browser DOM.

Search filters currently include:

```text
type:paragraph
type:block_quote
chapter:3
status:needs_review
marker:SOURCE_NEEDED
```

## Preview API

- `POST /projects/{project_id}/previews`
- `GET /projects/{project_id}/previews`
- `GET /previews/{preview_id}`
- `GET /previews/{preview_id}/file`

Preview jobs are handled by the same durable PostgreSQL worker as ingestion and exports. Heavy conversions remain serialized on the initial small VM.

## Browser recovery

The cockpit stores an unsaved block draft in `localStorage` before autosave. It displays:

- Saving…
- Saved
- Offline — changes kept locally
- Conflict detected — reload before saving
- Session expired — local drafts preserved

Local drafts are scoped to project and block UUID and cleared only after a successful version-bound command.

## Accessibility and responsive behavior

- Semantic navigation and landmarks
- Skip link
- Visible keyboard focus
- Severity text in addition to color
- Screen-reader save-status announcements
- Reduced-motion support
- Minimum touch targets
- Keyboard shortcuts:
  - `Ctrl/Cmd + S` save current editor
  - `Ctrl/Cmd + Z` undo
  - `Ctrl/Cmd + Shift + Z` redo
  - `Ctrl/Cmd + K` search
  - `Alt + Up/Down` move selected block
  - `Ctrl/Cmd + Enter` acknowledge an eligible selected review item

Desktop/tablet exposes full editing. Mobile prioritizes review, reading, small corrections, registry verification and downloads rather than pretending to offer desktop feature parity.

## Phase 2 acceptance path

1. Open an imported Phase 1 project.
2. Read deterministic readiness and persistent open findings.
3. Open a structural ambiguity.
4. Compare current block with immutable import.
5. Convert or edit the block through a command.
6. Resolve an ambiguous citation against a source.
7. Compare a quotation with the registry and verify it.
8. Complete missing metadata.
9. Undo and redo a prose edit.
10. Create and compare a checkpoint.
11. Generate an authoritative PDF preview.
12. Observe older exports become stale after editing.
13. Resolve the final underlying blocking issue.
14. Produce a version-bound final DOCX/PDF export.

## Deliberately excluded

- AI drafting and rewriting
- Supervisor invitations or permissions
- Real-time multi-user editing
- Full Word round-trip fidelity
- Billing
- Plagiarism/AI-detection scoring
- Institution administration
- Mobile full-editor parity

Those features must build on the command, snapshot, provenance and review contracts created here rather than bypassing them.
