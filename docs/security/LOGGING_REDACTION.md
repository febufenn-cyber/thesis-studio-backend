# Logging and Redaction Rules — As Implemented

Status: **review-preparation document**. This describes the logging and redaction behavior actually present in the code, with test evidence, plus the known gaps. It is not a claim that production log output has been independently audited — that sampling ("Production log review — Privacy owner") is listed as manual evidence in `docs/phase5/security-verification-matrix.md`.

## The rule

From `PHASE5-COMMERCIAL-OPERATING-CONTRACT.md` (Non-negotiable boundaries): "Logs may contain identifiers and hashes, but never thesis prose, quotations, full email addresses or private AI prompts." Restated as data-minimisation rule 1 in `docs/phase5/data-map.md`: "Do not log request bodies, query strings, thesis prose, quotations, AI prompts or full email addresses."

## What is never logged, and where that is enforced

The enforcement point for HTTP-layer logging is `JourneyTracingMiddleware` (`app/commercial/observability.py:54-115`). Its module docstring is the policy: "Logs contain route templates, hashes and operational identifiers only. Request bodies, query strings, emails, thesis text, quotations and AI prompts are never logged here." Concretely, the single per-request record contains only: timestamp, event name, `request_id`, `trace_id`, method, **route template** (not the concrete path with IDs where a route match exists), status, duration, response byte count, `client_hash`, and `release_sha` (`observability.py:100-113`). There is no body capture, no query-string capture, and no header capture beyond the sanitized inbound `x-request-id`/`x-trace-id` (validated to alphanumeric/`-_.` and truncated to 100 chars, lines 64-68).

Test evidence:

- `test_release_and_log_identity_are_safe` (`tests/test_phase5_unit.py:37-42`) — asserts the release-identity block contains no JWT material and that `opaque_identifier("student@example.edu")` is a 16-char value that is not the email itself. This pins the two building blocks every log record uses (`release_identity()` and `opaque_identifier()`, `app/commercial/observability.py:35-51`).
- `test_support_diagnostic_never_contains_thesis_text` (`tests/test_phase5_reliability.py:113-137`) — seeds a project whose abstract and chapter prose contain sentinel strings, generates a support diagnostic bundle, serializes it, and asserts the sentinels are absent and `bundle["privacy"]["manuscript_content_included"] is False`. This is the regression gate for the support/diagnostic surface.

Adjacent standing rules that keep secrets out of logs at the source: raw sign-in tokens are never stored, only SHA-256 hashes (`app/models/auth_token.py:41-42`); server-session tokens likewise (`token_hash`, `app/commercial/sessions.py:30-31`); repo policy forbids logging `Authorization` headers, JWTs, API keys, or full `messages[]` arrays and requires truncating user content to 100 chars in any log line (`CLAUDE.md`, "Never log secrets").

## Identifier hashing (PRIVACY_HASH_PEPPER)

Two hashing functions exist, and the difference matters for the review:

| Function | Location | Construction | Used for |
|---|---|---|---|
| `_privacy_hash` | `app/commercial/sessions.py:34-38` | SHA-256 over `effective_privacy_hash_pepper + "\x00" + value` | Session fingerprints: `application_sessions.user_agent_hash`, `ip_prefix_hash` (IPs are first truncated to a /24 (IPv4) or /56 (IPv6) prefix, `sessions.py:41-52`) |
| `opaque_identifier` | `app/commercial/observability.py:35-38` | Plain SHA-256, truncated to 16 hex chars, **no pepper** | Log-line identifiers, e.g. `client_hash` of the client IP in the journey record (`observability.py:110`) |

`PRIVACY_HASH_PEPPER` is an environment setting (`app/core/config.py:88`); when unset, `effective_privacy_hash_pepper` falls back to `JWT_SECRET` (`config.py:99-101`).

Two honest observations for the reviewer: first, `opaque_identifier` is unpeppered and truncated, so low-entropy inputs (IPv4 addresses, known email addresses) are brute-forceable offline by anyone holding the logs — acceptable for correlation IDs, but it should not be treated as anonymization; second, the pepper's fallback to `JWT_SECRET` couples log-pseudonymization strength to the auth signing secret and complicates independent rotation — production should set a distinct `PRIVACY_HASH_PEPPER`.

## Diagnostic bundle content policy

Support diagnostics are generated by `diagnostic_bundle` (`app/commercial/support.py:26-155`), which implements the runbook rule "Support diagnostics expose versions, checksums, states and object presence—not manuscript content" (`docs/phase5/data-map.md`, minimisation rule 3; `docs/runbooks/support-operations.md`, "Generate a metadata-only diagnostic bundle").

What the bundle contains: release identity; project workflow/version/profile metadata (`support.py:54-69`); job queue state with `error_class_present` as a **boolean** rather than the raw error text (`support.py:83`); export/preview/revision rows reduced to status, checksum, size, page count, and `has_storage_object` presence flags (`support.py:90-131`); AI activity as **status counts only** (`ai_run_counts`, lines 44-47, 132) — never message content; review/approval **counts** (lines 48-49). Every bundle carries a self-describing privacy block (`manuscript_content_included`, `source_or_quote_content_included`, `private_ai_content_included`, `full_email_included`, all `False`; lines 135-140) and writes a `support_actions` audit row with `content_accessed=False` and a justification (lines 142-153; model at `app/models/commercial.py:829-852`). Content access, when genuinely necessary, requires an explicit time-limited `support_access_grants` row with a consent note (`app/models/tenancy.py:262-282`) and the consent workflow in `docs/runbooks/support-operations.md` step 7.

## Gaps and review items

1. **Dev-mode email logging leaks identity and live secrets by design.** When `RESEND_API_KEY` is unset, `send_magic_link_email` logs the full recipient email plus the raw magic-link URL, and `send_otp_email` logs the email plus the OTP code, at WARNING level (`app/services/email_service.py:33-38, 102-106`). This is intentional for local development, but the guard is "API key unset," not `ENV != "production"` — a production misconfiguration with a missing key would degrade into logging live credentials instead of failing. Recommend an explicit production guard.
2. **Resend error responses are echoed into logs.** On API failure the first 500 chars of the response body are logged (`app/services/email_service.py:54-59, 131-136`); provider error bodies commonly echo the recipient address, which would put a full email address in the log. Recommend redacting or logging status + error code only.
3. **Free-text `error_message` columns are unredacted at rest.** `jobs.error_message` (`app/models/job.py:49`), `ai_runs.error_message` (`app/models/ai_run.py:50`), `exports.error_message`, `manuscript_revisions.error_message` store raw exception/stderr text in PostgreSQL. The support bundle already reduces this to a presence boolean, but anyone with DB access sees the raw text, and nothing prevents a parser or provider error from embedding a fragment of document content. Recommend a write-side sanitizer or an explicit review of what each producer writes.
4. **Proxy- and host-layer logs are out of app control.** Cloudflare (Tunnel/Access) and any uvicorn/PM2 access logging on the VM see full concrete URLs — including magic-link callback URLs with tokens in the query string (`magic_link_url_template`, `app/core/config.py:95-97`). The app's own journey log avoids query strings, but the review must confirm uvicorn access logs are disabled or route-templated in production and assess Cloudflare log retention. Token TTL (15 min) and single-use semantics bound, but do not eliminate, this exposure.
5. **`billing_events.payload` retains full webhook JSONB** (`app/models/commercial.py:318`) — excluded from card data by provider design (`docs/phase5/data-map.md`, "payload excludes payment-card details") but it is stored verbatim and is not subject to the logging redaction rules since it is a DB column; retention is a finance-policy decision (see `RETENTION_AND_DELETION.md`, open decision 1).
6. **No automated log-content scanner.** The privacy tests pin the two structured surfaces (journey record, support bundle); ad-hoc `log.info(...)` calls elsewhere in the app are constrained by convention (`CLAUDE.md`) but not by a lint/CI check. A grep-based CI guard for `email`, `content`, `payload` in logging calls would be cheap insurance.
